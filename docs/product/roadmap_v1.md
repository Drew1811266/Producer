# Producer V1 路线图

## 路线图原则
V1 的目标不是尽快堆功能，而是围绕一个模板、一套分层上下文图和一组稳定交互语法，跑通第一个可用闭环。路线图必须服从 `Phase 0 -> Module A -> 后续模块` 的顺序，不能跳过基础交互直接扩张成平台化产品。

## 当前基线：Phase 0 已落地部分
仓库当前已经具备 V1 起点，但还不具备完整可用性。

- 本地项目创建与打开已经存在。
- `.producer/`、`assets/`、`exports/` 的本地目录结构已经初始化。
- 声明式模板 `ecommerce_ad_v1` 已接入，并带有严格的 layer 与 node 白名单校验。
- 根 `Brief Canvas` 会在项目创建时先被 seed，当前只 seed 这一层。
- 最小全屏 workspace shell 已存在，说明画布优先的空间结构已确定。

## V1 核心阶段：Module A
Module A 负责把“能启动”推进到“能完成一次最小创作流”。

- 图中图导航与 camera：用户可以从根 `Brief Canvas` 出发，稳定进入 storyboard，再进入 shot lab。
- 节点渲染与选择：节点在画布中可见、可选、可聚焦，画布不再只是占位态。
- 单击抽屉编辑：选中节点后可在单一抽屉内编辑语义内容，不离开画布主视图。
- `Tab` 快速加点：在当前 layer 中按白名单创建合法节点，禁止越层创建。
- 节点持久化与白名单 enforcement：新增、编辑、关系变更都写入本地存储，并持续遵守模板约束。

Module A 完成后，Producer 才算具备第一个真实的“brief -> storyboard -> shot lab”创作闭环。

## V1 建议模块顺序
建议按下面顺序推进，而不是并行扩散功能面。

1. A. 项目目录与本地数据库结构
2. B. 画布相机与多层 graph 切换
3. C. 节点基础渲染与选中机制
4. D. 左侧节点内容抽屉
5. E. `Tab` 节点快速创建器
6. F. 节点持久化与层级白名单校验
7. G. 双击下钻与系统锚点生成
8. H. 缩略图缓存与媒体索引
9. I. 语义缩放与 LOD 渲染
10. J. 附件绑定、预览和节点关系显示

这个顺序不是功能罗列，而是约束顺序。没有 graph-of-graphs、白名单 enforcement 和高性能相机，后面的媒体、搜索和导出都会建立在错误模型上。

## V1 范围内的扩展方向
后续模块只能围绕已经验证的核心闭环外扩，而不是改写产品本体。

- 新模板：在 `ecommerce_ad_v1` 跑通后，再复制到其他垂直创作模板。
- 关系与评审增强：补齐 review、approved_from、variant_of 等语义关系的使用体验。
- 资产与结果沉淀：让 reference、still、video、result 与本地资产目录形成稳定绑定。
- 搜索与导出：围绕本地项目做检索、筛选、导出，而不是先做云端协作层。

这些扩展都必须复用同一套 canvas-first UI、单击抽屉、双击钻取、Tab 快加和语义关系模型。

## V1 明确不做
- 不做通用白板能力扩张。
- 不做在线多人协作平台。
- 不做看板式任务系统作为主界面。
- 不做素材库优先的内容管理产品。
- 不做 Houdini 风格的可执行节点流系统。

## 阶段验收口径
V1 是否成立，不看功能数量，看以下问题是否被同时回答。

- 用户能否从本地创建项目后立即落在根 `Brief Canvas`，并理解这是项目入口。
- 用户能否通过单击、双击和 `Tab` 完成主要编辑与导航，而不依赖复杂面板。
- 用户能否在全局与局部之间顺畅平移缩放，并通过语义缩放读懂项目结构。
- 用户能否明确感知每一层允许什么节点、为什么不能乱放。
- 用户能否把关系理解为创作语义，而不是执行流程。
- 用户能否在 Shot Lab 中把提示词、静帧、视频、参考、评审和结果组织成一个可追溯的镜头工作台。
- 团队能否在不破坏产品边界的前提下，为后续模块继续扩展。

如果以上条件不成立，就不应该继续扩张模块数量，而应回到 Phase 0 + Module A 把核心闭环打磨完整。
